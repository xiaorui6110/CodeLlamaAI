# CodeLlamaAI Phase 1 验收总结与下一步计划

## 一、Phase 1 验收结论

截至当前版本，CodeLlamaAI 的 Phase 1 目标已经基本完成，可以视为通过验收。

Phase 1 原始目标是：让插件在 IDEA 中稳定完成基础问答和代码解释，并从“最小聊天骨架”升级为“可用插件”。

结合当前实现结果，结论如下：

- 已完成正式的 Settings 页面
- 已完成多轮会话式 Tool Window
- 已完成当前文件与选区上下文接入
- 已完成 PromptBuilder 与结构化请求对象
- 已完成基础错误处理与状态反馈
- 已完成流式输出
- 已完成请求取消
- 已完成常见 Markdown 语法展示

因此，Phase 1 的核心验收标准已经满足。

## 二、Phase 1 已交付能力

当前插件已经具备以下能力：

### 1. 基础配置能力

- 可在 Settings 中配置 Ollama 地址
- 可配置系统提示词
- 可配置上下文收集选项
- 可配置上下文长度限制

### 2. 本地模型接入能力

- 可拉取本地 Ollama 模型列表
- 模型下拉框可展示模型大小信息
- 未选择模型时可自动优先选择更小模型
- 可避免多轮对话过程中模型被错误重置

### 3. 会话与交互能力

- 支持多轮消息展示
- 支持流式回复
- 支持中断当前请求
- 支持基础失败提示
- 支持 Markdown 结果展示

### 4. IDEA 上下文能力

- 可读取当前活动文件
- 可读取当前编辑器选区
- 可读取光标附近代码片段
- 可将上述上下文拼装进模型请求

### 5. 工程结构能力

- 已具备 `context / prompt / chat / settings / toolwindow / ollama` 的基础模块拆分
- 请求链路已经不再是单纯的字符串拼接
- 后续继续扩展动作入口和任务类型时，已有较稳定基础

## 三、Phase 1 遗留问题

虽然 Phase 1 已经达标，但仍有一些尚未打磨完成的部分，这些问题更偏向“体验优化”，不影响当前阶段验收：

- Tool Window 整体视觉样式还比较基础
- Markdown 渲染目前是轻量实现，覆盖常见格式但不是完整规范支持
- 代码块展示还没有复制按钮、专门高亮或折叠能力
- 会话历史还没有持久化
- 暂时没有 Editor Popup、右键菜单、快捷动作等高频入口
- 暂时没有针对代码任务的结构化操作入口

这些内容更适合作为下一阶段或 Phase 1.5 的工作。

## 四、为什么可以结束 Phase 1

判断一个阶段是否结束，不应看“还有没有可以继续优化的地方”，而应看“该阶段的核心目标是否已经达成”。

Phase 1 的重点不是：

- 做漂亮界面
- 做大量功能入口
- 做自动改代码

Phase 1 的重点是：

- 插件能跑起来
- 基础请求链路稳定
- 能读取 IDE 上下文
- 本地模型可实际完成一次可用对话
- 用户在模型较慢时依然有反馈

这些点现在都已经满足，所以 Phase 1 可以收尾。

## 五、下一步建议：进入 Phase 2 前的过渡阶段

虽然可以直接进入 Phase 2，但更合理的做法是先安排一个较短的“Phase 1.5”，专门处理高价值但低风险的体验增强，再开始真正的动作型能力扩展。

建议将下一步拆成两个层次：

### 方案 A：先做 Phase 1.5 体验增强

目标：让当前聊天工作流更顺手、更像一个正式插件。

建议优先做：

1. 会话消息样式统一
2. 代码块视觉优化
3. 代码块复制按钮
4. 会话区滚动体验优化
5. 输入区与结果区布局微调
6. 更好的状态提示文案
7. Markdown 展示细节修正

这一阶段的特点是：

- 风险低
- 不会破坏主链路
- 能明显提升主观体验
- 有利于后续继续扩展更多入口

### 方案 B：直接启动 Phase 2 核心能力

目标：让插件进入真实开发工作流，而不只是 Tool Window 聊天。

建议首批能力：

1. Explain Code 动作
2. Generate Tests 动作
3. 编辑器选区右键入口
4. Action 命令入口
5. 任务类型化的 Prompt 模板

这一阶段的重点是把“聊天能力”变成“开发动作”。

## 六、推荐的下一阶段实施顺序

综合当前状态，推荐顺序如下：

### Step 1：做一轮轻量体验打磨

建议范围：

- Tool Window 基础视觉优化
- Markdown 展示细化
- 代码块展示体验优化

原因：

- 当前主链路已经打通
- 再继续往深处扩展功能前，先把已有入口做顺
- 用户会更愿意持续使用和反馈

### Step 2：引入第一个真正的编辑器动作

建议优先做：

- Explain Code

原因：

- 技术路径最清晰
- 能直接复用现有上下文收集与 PromptBuilder
- 风险比自动生成代码低
- 最容易建立“插件不是聊天框，而是 IDE 工具”的感觉

### Step 3：引入第二个高价值动作

建议优先做：

- Generate Tests

原因：

- 用户价值高
- 与 Explain Code 一起能形成比较明显的插件定位
- 可作为后续更多代码任务动作的模板

## 七、建议的近期任务拆分

如果按最近一轮开发安排，建议拆成下面几个任务：

### 任务 1：界面与展示优化

- 调整 Tool Window 布局层次
- 优化消息卡片样式
- 优化代码块与 Markdown 展示
- 增加复制代码按钮

### 任务 2：任务类型抽象

- 为不同任务定义基础类型
- 将 Explain Code 与普通聊天区分开
- 为后续 Generate Tests 预留入口

### 任务 3：编辑器动作入口

- 新增 `AnAction`
- 支持对选区执行 Explain Code
- 支持从编辑器上下文触发请求

### 任务 4：测试补强

- 为 MarkdownRenderer 增加单元测试
- 为 PromptBuilder 和上下文逻辑补充边界测试
- 为核心错误场景增加回归保护

## 八、最终建议

当前最合适的策略不是立刻大规模扩展，而是：

先做一轮轻量体验打磨，再进入 Phase 2 的动作化扩展。

推荐下一步主线为：

1. 统一界面与消息展示体验
2. 实现 Explain Code 动作
3. 实现 Generate Tests 动作
4. 逐步减少用户对 Tool Window 手动输入的依赖

## 九、阶段判断

当前项目状态可以定义为：

**Phase 1 已完成，准备进入 Phase 1.5 / Phase 2 过渡阶段。**

后续如果继续推进，建议优先围绕“体验优化 + 第一个编辑器动作”展开，而不是继续扩展聊天框本身。
