# CodeLlamaAI 项目规划与功能扩展方向

## 1. 当前项目状态判断

结合现有文档与源码，这个 IDEA 插件目前处于“最小可用骨架”阶段，已经验证了本地 Ollama 接入和基础 UI 链路，但距离“可持续使用的 IDE 内 AI 编程助手”还有明显差距。

当前已经具备的能力：

- 通过 Tool Window 提供基础聊天入口
- 持久化保存 Ollama 地址、模型名、系统提示词
- 从本地 Ollama 拉取已安装模型列表
- 发送单轮 Prompt 并显示模型返回结果
- 用 `chat / ollama / settings / toolwindow` 做了初步分层

当前明显的边界和限制：

- 交互还是“通用聊天框”，没有 IDE 场景能力
- 只有单轮请求，没有会话上下文管理
- 没有接入编辑器选区、当前文件、工程结构等上下文
- 没有流式输出、取消请求、重试、历史记录
- 还没有 Settings 页面、Action、右键菜单、编辑器内操作
- 代码生成结果不能直接作用于编辑器，也没有 diff/预览能力
- 测试仍以本地联通性验证为主，缺少核心业务回归保护

结论：

这个项目的核心问题已经不是“能否连上 Ollama”，而是“如何从聊天面板升级为真正嵌入 IDEA 工作流的开发助手”。

## 2. 产品定位建议

建议把 CodeLlamaAI 的定位收敛为：

**一个以本地模型和隐私优先为特点、面向 IntelliJ IDEA 日常编码场景的轻量 AI 助手。**

这个定位有几个好处：

- 与云端 Copilot 类产品形成差异化，突出“本地可控”
- 与当前技术栈一致，不需要一开始就做复杂的多模型平台
- 能够围绕 IDEA 场景逐步增强，而不是泛化成普通聊天客户端

建议优先覆盖的核心用户场景：

1. 解释当前代码
2. 基于选区做重构建议
3. 基于报错或日志做问题分析
4. 生成单个方法、测试用例或注释
5. 对当前文件进行审阅和改进建议

不建议前期优先做的方向：

- 大而全的 Agent 工作流
- 自动执行命令或自动修改多个文件
- 复杂的多模型路由平台
- 与远程服务深度绑定的账号体系

原因很直接：当前项目的基础设施还不足以支撑高自动化能力，过早扩张只会让插件变成不稳定的“聊天壳”。

## 3. 功能扩展方向

### 3.1 第一优先级：把聊天能力变成 IDE 能力

这是最关键的一步，建议优先补齐：

- 当前文件上下文注入
- 编辑器选区上下文注入
- 光标附近代码片段提取
- 会话级上下文管理
- 更清晰的 Prompt 构建机制

建议新增模块：

- `context`
  - 负责从 IDEA 中提取当前文件、选区、语言类型、光标位置、类/方法名等信息
- `prompt`
  - 负责统一拼装系统提示词、用户输入、代码上下文、任务模板
- `session`
  - 负责管理会话历史、消息结构、最近一次上下文来源

完成这一步后，插件才会从“手动复制代码去问模型”进化为“在 IDE 环境里直接使用模型”。

### 3.2 第二优先级：围绕高频操作提供入口

建议增加而不是继续堆积单一 Tool Window：

- 编辑器右键菜单
- `AnAction` 命令入口
- 选区快捷动作
- 工具栏常用操作

适合首批落地的命令：

- Explain Code
- Refactor This
- Generate Tests
- Fix Error
- Summarize File

这些能力的价值高，而且实现方式相对清晰，能直接复用上下文收集和 PromptBuilder。

### 3.3 第三优先级：提升交互体验

如果仍保留 Tool Window，它应逐步升级为“会话中心”，而不是简单输入框：

- 流式输出
- 请求取消
- 重试最近一次请求
- 多轮消息展示
- 消息角色区分
- 渲染代码块
- 一键复制代码

进一步可以考虑：

- 会话历史保存
- 按项目维度保存最近会话
- 常用 Prompt 模板

### 3.4 第四优先级：让回答可操作

这是插件从“建议工具”走向“生产力工具”的关键：

- 将代码块一键插入编辑器
- 预览变更 diff 后再应用
- 针对选区执行替换
- 将解释结果插入注释或文档

建议在这一阶段再引入：

- `codeaction`
  - 负责把模型输出转换为 IDE 中可执行的动作
- `diff`
  - 负责变更预览与确认

如果没有 diff/确认机制，就不适合过早开放自动修改能力。

## 4. 技术架构演进建议

### 4.1 保留现有分层，但补上中间层

当前 `toolwindow -> chat service -> ollama gateway` 的结构方向是对的，但还不够支撑复杂功能。建议演进为：

```text
ui(action/toolwindow/settings)
  -> application service
  -> context / prompt / session / codeaction
  -> model gateway
  -> persistence
```

更具体一点：

- `ui`
  - Tool Window、Settings、Action、Popup
- `application`
  - 编排一次 AI 请求的完整流程
- `context`
  - 收集 IDE 上下文
- `prompt`
  - 统一生成 Prompt
- `session`
  - 管理会话与消息
- `model`
  - 对接 Ollama，后续可抽象多模型接口
- `persistence`
  - 保存配置、会话、模板等数据

### 4.2 尽早抽象统一请求对象

当前 `sendPromptAsync(String prompt)` 过于简单。建议后续改成显式请求对象，例如：

```text
AiRequest
- taskType
- userInput
- systemPrompt
- contextItems
- modelName
- stream
```

这样可以避免后续新增能力时持续改方法签名，也更利于测试。

### 4.3 模型返回值也需要结构化

目前 `ChatResult` 只有 `success + content`，后续至少应扩展出：

- message content
- error message
- finish reason
- token/statistics（如果底层可拿到）
- raw blocks / code blocks
- action suggestions

如果后续要做代码插入、diff 应用、消息渲染，结构化结果是必须的。

### 4.4 先支持流式，再谈高级交互

用户对插件体验最直观的判断来自“是否有反馈”。所以技术上建议优先解决：

- 流式响应
- 后台任务状态
- 取消机制
- UI 线程与后台线程边界

这部分比继续增加单轮聊天功能更重要。

## 5. 建议的阶段路线图

### Phase 1：从聊天面板升级为可用插件

目标：让用户在 IDEA 里稳定完成基础问答和代码解释。

建议交付：

- 正式 Settings 页面
- 会话消息列表 UI
- 流式输出
- 当前文件 / 选区上下文接入
- PromptBuilder
- 基础错误处理和状态提示

验收标准：

- 用户无需手动复制大量代码到输入框
- 用户能感知请求进行中和失败原因
- 至少有一个“解释代码”场景稳定可用

### Phase 2：从可用插件升级为高频工具

目标：让插件进入真实开发工作流。

建议交付：

- Editor Popup / 右键菜单 / 快捷动作
- Explain / Refactor / Generate Tests / Fix Error 四类动作
- 会话历史保存
- 常用任务模板
- 更稳定的上下文截断策略

验收标准：

- 用户能在选区上直接触发 AI 操作
- 高频任务不需要重新组织描述
- 插件在中等规模项目中依然可响应

### Phase 3：从建议工具升级为半自动协作工具

目标：让模型输出可以安全地作用于代码。

建议交付：

- 代码块一键插入
- 生成结果 diff 预览
- 选区替换或新文件草稿生成
- 审阅结果结构化展示

验收标准：

- 用户可以在确认后应用模型生成内容
- 不会因为回答格式波动导致误改代码
- UI 对“建议”和“执行”有明确边界

## 6. 测试与质量建设建议

当前项目的测试保护比较薄，建议尽早建立下面三层：

1. 单元测试
   - PromptBuilder
   - 上下文裁剪与拼装
   - Ollama 模型列表解析
2. 集成测试
   - 本地 Ollama 联通
   - 关键服务调用链
3. 插件行为测试
   - Tool Window 基础交互
   - Action 在有/无编辑器上下文下的行为

建议把依赖本地 Ollama 的测试与纯逻辑测试分开，避免所有测试都被外部环境拖住。

## 7. 近期最值得做的开发清单

如果只考虑接下来 2 到 4 周的投入，建议优先做以下事项：

1. 新增正式 Settings 页面，Tool Window 不再承担主要配置职责
2. 设计 `AiRequest / AiResponse / ChatMessage` 等基础领域对象
3. 引入 `ContextCollector` 和 `PromptBuilder`
4. 支持当前文件与选区上下文问答
5. 将 Tool Window 改造成多轮会话视图
6. 增加流式输出与取消请求
7. 新增首批编辑器动作：Explain Code、Generate Tests
8. 为 Prompt 和上下文处理补齐单元测试

这是当前阶段性价比最高的一条路线，因为它同时补了“可用性、IDE 集成、架构扩展性”三件事。

## 8. 风险与设计原则

后续演进时建议坚持以下原则：

- 先做稳定的单文件、单选区能力，再做跨文件智能体
- 先让输出可读、可取消、可重试，再追求复杂自动化
- 先结构化请求与响应，再增加功能入口
- 先做“用户确认后应用”，不要直接自动改代码
- 先围绕 IDEA 场景设计，不要退化成普通聊天窗口

主要风险：

- 本地模型能力和速度不稳定，导致体验波动
- Prompt 和上下文策略过早耦合在 UI 中，后续难维护
- 功能入口越来越多，但没有统一任务模型
- 直接做代码修改却没有安全确认链路

## 10. 总结

CodeLlamaAI 当前最合理的发展路径，不是继续扩展“聊天框能问什么”，而是逐步建立：

- 面向 IDEA 的上下文收集能力
- 面向任务的 Prompt 与请求模型
- 面向开发动作的 UI 入口
- 面向代码变更的安全应用机制

只要先把这四个基础打稳，后续无论接入更多模型、做更复杂的代码生成，还是演进到更强的本地 AI 编程助手，都会顺很多。
